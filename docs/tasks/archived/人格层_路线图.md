## 模块职责

人格层是整个项目的核心，也是最需要持续打磨的部分。

- **负责**：角色配置加载、情绪状态管理、prompt 组装、输出约束（禁用词/口癖）
- **不负责**：UI 展示、记忆存储、感知采集、LLM 调用
- **约束**：所有其他层只向人格层提供输入，最终输出由人格层统一决定

### 编码边界

- 人格层内部尽量保持纯逻辑，不直接读写文件、不直接调第三方 SDK。
- 人格层只接收 `CharacterProfile`、`EmotionState`、`MemoryContext`、`PerceptionContext` 这类结构化对象。
- `prompt_builder` 只负责组装 prompt，不做记忆检索、事件识别、日志写入。
- 输出后处理如禁用词替换、口癖约束应集中在单一模块，避免分散在 CLI 或 UI。

## Phase 1：人格层定型

目标：把人格层从"能跑"提升到"稳定可控、可验证、可维护"。

### 1.1 角色配置 Schema 固化

设计目标配置格式（参考设计文档）：

```yaml
# characters/<name>/personality.yaml
name: ""
version: ""
schema_version: "1"   # 为 Phase 6 角色包兼容预留

personality:
  core_fear: ""       # 核心恐惧/弱点
  surface_trait: ""   # 外在表现
  hidden_trait: ""    # 内在真实

behavior_rules:
  - ""  # 具体行为约束，越精确越好

proactive_style:
  idle_too_long: ""
  user_working_late: ""
  user_gaming: ""

forbidden_words: []
verbal_habits: []

# 每种情绪在 prompt 中展示的自然语言描述
mood_expressions:
  normal: ""
  happy: ""
  angry: ""
  shy: ""
  cold: ""

emotion_triggers:
  happy: []
  angry: []
  shy: []
  cold: []
```

任务：

- [ ] 用 Python dataclass 或 pydantic 定义完整 schema
- [ ] 写 `validate_character(config)` 函数，在加载时运行
- [ ] 所有字段加默认值，避免新角色缺可选字段报错
- [ ] 区分"必填字段缺失"与"可选字段未配置"两类错误，报错信息可定位到 YAML 路径
- [ ] 保留 `schema_version` 字段，为 Phase 6 的角色包兼容预留升级空间

### 1.2 情绪状态机增强

首版只保留最少可验证状态：

```python
class EmotionState:
    mood: str          # normal / happy / angry / shy / cold
    persist_count: int # 情绪持续轮数，用于 decay
```

`affection`、`energy`、`last_interaction` 等在桌面阶段（Phase 4）再引入。

> **当前代码问题**：`emotion.py` 的 `update(user_input, triggers)` 把"触发词匹配"和"状态跃迁"耦合在一起，导致无法单独测试事件检测逻辑。Phase 1 必须拆分。

任务：

- [ ] 固化 `EmotionState` 数据结构，只保留 `mood` 和 `persist_count`
- [ ] 从 `update()` 中拆出 `detect_event(user_input, triggers) -> str | None`，返回情绪名称或 None，不改状态
- [ ] `update()` 改为接受已检测到的事件名（`str | None`），只负责状态跃迁和衰减
- [ ] `decay()` 每轮对话后调用，`persist_count` 归零时 mood 回到 `normal`
- [ ] 写单元测试：单独测试 `detect_event`（输入→事件），单独测试 `update`（事件→状态）

### 1.3 Prompt 构建集中化

目标签名（Phase 0 先实现不含记忆/感知的版本，Phase 2/3 补充）：

```python
# src/personality/prompt_builder.py

def build_system_prompt(ctx: PromptContext) -> str:
    """唯一入口，所有 prompt 拼接只在此处发生。"""
    ...
```

`PromptContext` 定义见总览 §4.2，Phase 0 起就用这个签名，避免参数膨胀后再改接口。

任务：

- [ ] 将 `build_system_prompt(config, state)` 重构为接收 `PromptContext`
- [ ] prompt 模板按角色分区：基础人格 / 当前情绪描述 / 行为约束 / 记忆上下文（可选）/ 感知上下文（可选）
- [ ] 增加 prompt 长度估算（按字符数粗估），超过阈值时记录警告日志
- [ ] `build_system_prompt` 只接收已准备好的上下文，不负责自己读取记忆或感知
- [ ] 写单元测试：给定角色和情绪，验证 prompt 包含必要字段且不包含禁用词汇

### 1.4 人格稳定性日志指标

任务：

- [ ] 在会话日志中记录每轮的情绪状态（写入 JSONL 的 `emotion` 字段）
- [ ] `--replay` 模式下能展示每轮情绪变化轨迹
- [ ] 提供一个简单脚本 `tools/review_emotion.py`，从日志中统计情绪分布

### 验收

- 明确触发词输入后，情绪变化在 1-2 轮内可预测
- `build_system_prompt` 单元测试通过

## Phase 2 之后：台词 RAG（后置）

台词检索增强（RAG）是人格层的锦上添花项，在人格稳定之后再引入。

```
角色台词数据 → 按情绪标签分类 + embedding → ChromaDB 本地存储
用户输入 → embedding → 检索同情绪标签台词 → 注入 prompt 作为 few-shot 示例
```

前置条件：

- 有足够干净的台词库（因角色而异，需单独整理）
- 长期记忆结构已稳定（避免与 ChromaDB 结构冲突）
- few-shot 数量保持极少，只作为风格校准，不能压过角色基础设定

任务（后置，Phase 3 之后）：

- [ ] 整理目标角色的台词数据，按情绪标签分类
- [ ] 用 `text-embedding-3-small` 向量化台词
- [ ] 接入 ChromaDB，按情绪标签分 collection 存储
- [ ] 在 `prompt_builder.py` 中增加 `rag_context` 可选注入点
- [ ] 验证 RAG 开启/关闭对输出风格的影响
