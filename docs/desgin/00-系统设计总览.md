## 流程
1. CLI 模式由 `main.py` 读取 `.env`，实例化 `ConversationService`，可选启用感知层。
2. 桌面模式由 `src-tauri/src/main.rs` 启动 sidecar、托盘和窗口；前端只通过 HTTP / WebSocket 调用 sidecar。
3. sidecar 由 `src/api/app.py` 在 lifespan 中创建全局 `ConversationService` 与 `CompanionRuntime`。
4. `CompanionRuntime` 启动后台 async loop，每轮先查询 `RoutineReminderService` 中已到期的 reminder，再按可用性采集并隐私过滤 `PerceptionContext`，统一交给 `ProactiveScheduler` 和策略层决定是否主动介入。
5. 若主动行为为 `expression`、`short` 或 `full`，runtime 通过 `/stream` 共用的 WebSocket 连接管理器广播 `proactive` 帧给主窗口；`full` 级别优先尝试 LLM，失败时降级到模板短句或表情。reminder 场景只把 `reminder_id` 和标题摘要写入主动日志，不写完整备注文本。
6. `/chat` 或 `/stream` 收到输入后调用 `ConversationService.handle_turn()` 或 `handle_turn_stream()`。
7. `ConversationService` 在 `_prepare_turn()` 中更新情绪、读取 `MemoryContext`、读取最近 `PerceptionContext`、生成关系摘要，并调用人格层组装 system prompt。
8. 能力层根据 `.env` 创建的 LLM client 执行对话；主动陪伴设置从 `data/runtime/proactive/settings.json` 读取，`.env` 只提供初始默认值。
9. `_finalize_turn()` 追加工作记忆、记录 token、检查禁用词、写入会话日志、更新 `RelationshipState`，并按轮数阈值触发自动记忆结算或重置空闲结算计时器。
10. 记忆层只结算自上次成功持久化后新增的消息片段；CLI 退出、角色切换、idle 超时、FastAPI shutdown 和桌面端退出前 flush 都走同一条结算路径，当前以 JSON 记录仓库为权威存储。
11. `/state`、`/health` 和 `/admin/*` 从当前服务实例、runtime、本地存储或 `.env` 读取状态返回给前端。
12. 任何可选能力初始化失败时，系统保持聊天主链路可用，并通过日志、管理界面或降级状态暴露问题。

## 约束与规则
- sidecar 运行态由 `ConversationService` 与 `CompanionRuntime` 共同组成；前者负责普通对话，后者负责后台主动陪伴。
- 前端主窗口和管理界面不得直接读角色目录或数据目录，只能消费 sidecar 输出。
- 管理界面允许在普通浏览器运行，涉及 Tauri API 的调用必须先检测运行时能力。
- 关系、记忆、感知只作为 prompt 的结构化输入，不允许改写角色身份。
- 若未来引入向量数据库，它只能作为记忆检索索引层挂在记忆仓库之上，不能直接替代 `facts.json` 的权威记录语义。
- 可选模块失败时优先降级，不允许阻断 `/chat`、日志记录或文本输入。
- 当前 sidecar 是单服务实例；角色切换通过替换服务实例完成，不支持多角色并发会话。
- 主动陪伴设置必须以本地持久化文件为权威来源；`.env` 中的 `KOKORO_ENABLE_PERCEPTION` 只用于首次启动默认值，不再作为常规操作入口。
- 感知隐私设置必须以 `data/runtime/perception/privacy_settings.json` 为权威来源；感知审计只能保存过滤后的摘要。
- reminder 由本地 `RoutineReminderService` 维护；一次性提醒到期后进入 `pending_ack`，周期提醒到期后推进下一次 `due_at`。
- 感知采集器初始化失败时，只禁用 perception-derived 信号，不阻断 reminder 场景的主动广播。
- AIRI 合并修正(追加:2026-04-28): Kokoro 保持 Python sidecar + Vue/Tauri 架构，不迁移到 AIRI monorepo 或 pnpm workspace；只吸收角色卡、表现事件、TTS 队列、NotifyEvent、结构化记忆、DisplayRuntime、插件事件和 computer-use 安全边界。
- AIRI 合并修正(追加:2026-04-28): Phase A 必须先完成角色卡 v2、表现层事件协议、TTS pipeline 和 NotifyEvent 主动入口；Phase B 再完成长期记忆治理；Phase C 再完成 VAD/STT 与 DisplayRuntime；Phase D 才允许开放插件协议和 computer-use 原型。

## 验收标准
- 执行 `python main.py --debug`；程序启动并进入交互循环，且在发送消息后输出回复和调试信息。
- 启动 sidecar 后请求 `/health`；返回 `status=ok` 且包含 `sidecar`、`llm`、`character_resources`、`tts` 四个状态块。
- 请求 `/state`；返回当前角色、情绪、记忆计数、关系快照和会话 token 统计。
- 在主窗口保持连接但用户不发送消息，满足主动条件后；收到 `type=proactive` 的 WebSocket 帧，且不增加普通 `turn`。
- 请求 `/admin/reminders` 创建一条已到期 reminder，并保持主窗口连接；runtime 产生 `scene=reminder` 的 proactive 记录，随后用户反馈可改变 reminder 状态。
- 配置感知标题黑名单并采集匹配窗口；普通 prompt、主动日志和感知审计均不包含原始窗口标题。
- 在桌面模式触发角色切换事件；主窗口状态重置为新角色，sidecar 返回新的 display 配置。
- 连续对话达到自动结算阈值，或关闭带感知的 CLI 会话；`data/memories/<character_id>/summaries.jsonl` 新增一条摘要或在摘要生成失败时保持原文件不损坏。
- 检查 Phase A 相关实现；角色切换后人格、声线、身体资源同时来自角色卡绑定信息，且普通聊天、主动提醒和 TTS 播放共用表现事件链路。
- 检查架构目录；新增 AIRI 合并能力不得引入新的 monorepo 根目录或绕过 sidecar 的前端直连 provider 调用。
