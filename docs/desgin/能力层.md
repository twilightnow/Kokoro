# 能力层

当前能力层非常聚焦，核心就是一件事：统一封装 LLM 调用。

## 当前模块

- `src/capability/llm.py`

对上层暴露两个核心对象：

- `LLMClient`
- `LLMResult`

应用层不需要关心具体 provider SDK，只拿统一的 `chat()` 结果。

## 当前支持的 provider

### CLI 模式

- `claude-cli`
- `gemini-cli`
- `codex-cli`

特点：

- 走本地已登录 CLI
- 一次性对话
- 不返回真实 token 用量时，usage 为 0 或空

### API Key 模式

- `anthropic`
- `openai`
- `gemini`
- `deepseek`
- `openrouter`
- `copilot`

这些 provider 通过统一工厂 `create_llm_client()` 创建。

## 当前设计重点

### 统一返回结构

无论底层 provider 是什么，最后都收敛成：

- `text`
- `model`
- `provider`
- `input_tokens`
- `output_tokens`
- `finish_reason`

### provider 选择策略

- 优先读取 `LLM_PROVIDER`
- 若只配置了一个可用 key，则自动选择
- 配置冲突时直接报错

### 配置边界

- provider 解析与鉴权都留在能力层内部
- 上层不关心 `base_url`、headers、CLI 命令差异

## 当前没有做的事

能力层当前还没有：

- 真正的流式 token 输出
- TTS
- ASR
- embedding 接口
- 视觉理解接口

因此能力层文档应该继续以 “LLM 抽象层” 为主，不要写成“大而全工具层”。

## 保持简单的约束

- 上层只依赖 `chat(system_prompt, messages)`
- provider 差异全部收敛在 `llm.py`
- 新增 provider 只改能力层，不改应用层协议

短期内这就是最稳的做法。等出现第二种稳定能力，比如 TTS，再单独拆新模块，而不是把所有能力继续堆进一个文件。
