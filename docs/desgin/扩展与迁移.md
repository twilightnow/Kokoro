# 扩展与迁移

当前代码已经具备“可扩”的基础，但还没有正式的角色包或插件系统。这份文档只描述现状和合理的扩展方向。

## 当前可扩展点

### 多角色

当前角色组织方式：

```text
characters/
  asuka/
    personality.yaml
  rei/
    personality.yaml
```

前后端都基于目录名做角色切换：

- Tauri 读取 `characters/` 下的子目录名
- sidecar 根据 `characters/<name>/personality.yaml` 重建服务实例

这已经能支持简单多角色，不需要先引入更复杂的角色包格式。

### 数据目录

当前默认数据目录来自：

- 环境变量 `KOKORO_DATA_DIR`
- 否则默认 `./data`

当前实际持久化内容主要有：

- `data/memories/<character_id>/summaries.jsonl`
- `data/memories/<character_id>/facts.json`
- `data/memories/<character_id>/truncation.log`
- `data/logs/*.jsonl`

这意味着当前项目已经具备基础迁移能力：角色目录和 `data` 目录一起迁走即可。

## 当前尚未落地的扩展

还没有正式实现：

- `manifest.yaml`
- `.charpkg`
- 角色安装器
- 插件扫描
- 工坊市场

所以短期内不要为了这些未来能力提前复杂化目录结构。

## 推荐的扩展顺序

### 第一阶段

继续保持当前简单结构：

- 一个角色一个目录
- 每个角色至少一个 `personality.yaml`
- 数据按角色隔离

### 第二阶段

如果角色资源开始增多，再考虑扩成：

```text
characters/
  asuka/
    personality.yaml
    assets/
    preview.png
```

### 第三阶段

只有在确实需要分享和安装时，再引入：

- `manifest.yaml`
- 压缩包格式
- 导入导出命令

## 迁移建议

当前项目最简单、最稳的迁移方式就是备份两个目录：

- `characters/`
- `data/`

只要这两个目录还在，角色配置和大部分用户数据都能跟着走。

## 保持模块化的约束

- 角色资源扩展不应影响 `ConversationService` 接口
- 数据目录变更只应影响存储层，不影响人格层和 UI 层
- 新的角色元数据优先放独立文件，不污染 `personality.yaml`

扩展的重点不是“提前设计一切”，而是保证以后要加东西时，不需要推翻现在已经稳定的边界。
