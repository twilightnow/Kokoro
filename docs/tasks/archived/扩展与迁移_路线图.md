# 扩展与迁移路线图

← [返回总览](开发路线图.md)

设计参考：[扩展与迁移设计](../../desgin/扩展与迁移.md)

---

## 模块职责

- **负责**：角色包格式标准化、用户数据目录统一、导入导出迁移、插件接口设计
- **时机**：Phase 6，在前五个阶段稳定后才启动
- **约束**：首版只需本地目录加载单角色，扩展机制全部后置

### 编码边界

- 迁移和扩展优先建立稳定格式，不急着做复杂执行能力。
- 首版角色包和插件只允许接入受控目录与受控元数据，避免一开始就开放任意代码执行。
- 数据目录规划要与运行目录解耦，项目源码目录不应承载用户长期数据。

## Phase 0：目录结构与数据根目录（必须 Phase 0 完成）

虽然完整扩展机制后置，但数据根目录必须在 Phase 0 就确立，否则 Phase 2 记忆落盘会硬编码路径，Phase 6 迁移成本极高。

### 任务清单

- [ ] 引入 `KOKORO_DATA_DIR` 环境变量，默认值 `./data`
- [ ] 在 `src/config.py`（或等价配置模块）中统一计算所有数据路径：

  ```python
  import os
  from pathlib import Path

  DATA_DIR = Path(os.environ.get("KOKORO_DATA_DIR", "data"))
  LOGS_DIR = DATA_DIR / "logs"
  MEMORIES_DIR = DATA_DIR / "memories"
  CHARACTERS_DIR = Path("characters")  # 角色定义跟随代码仓库，不放到 data
  ```

- [ ] 所有模块（`SessionLogger`、`SummaryMemory`、`LongTermMemory`）通过导入这些常量确定路径，不各自拼字符串
- [ ] `logs/` 按会话 ID 命名（现有实现已做），不混入角色数据
- [ ] `.env.example` 中加入 `KOKORO_DATA_DIR` 注释说明

### Phase 1 补充

- [ ] 确认角色目录结构为 `characters/<name>/personality.yaml`，不改动 `src/`
- [ ] 记忆数据存放路径提前约定（Phase 2 实现时直接按规范落盘）：`$KOKORO_DATA_DIR/memories/<character_id>/`

## Phase 2：记忆路径规范落盘

在实现记忆层时，直接按迁移友好的路径落盘（路径由 `src/config.py` 提供，不在记忆模块内拼接）。

### 任务清单

- [ ] 摘要记忆：`$KOKORO_DATA_DIR/memories/<character_id>/summaries.jsonl`
- [ ] 长期记忆：`$KOKORO_DATA_DIR/memories/<character_id>/facts.json`
- [ ] 路径通过 `src/config.py` 中的常量或工厂函数管理，记忆模块不出现路径字符串拼接
- [ ] 确认所有路径均基于 `KOKORO_DATA_DIR`，支持未来迁移

## Phase 6：扩展与迁移完整实现

### 6.1 角色包格式标准化

目标角色目录结构：

```
characters/
└── asuka/
    ├── manifest.yaml        # 包元数据
    ├── personality.yaml     # 人格配置
    ├── live2d/              # Live2D 模型文件（可选）
    │   ├── asuka.model3.json
    │   └── ...
    ├── voice/               # 语音素材（可选，用于 TTS 克隆）
    │   └── samples/
    ├── corpus/              # 台词库（可选，用于 RAG）
    │   └── lines.jsonl
    └── preview.png          # 预览图（可选）
```

`manifest.yaml` 结构：

```yaml
id: "asuka-langley-tv"
name: "惣流·明日香·兰格雷"
version: "1.0.0"
author: ""
description: ""
license: "CC BY-NC-SA"
requires:
  platform: ">=0.1.0"
  live2d_model: true
  tts_voice: true
```

任务：

- [ ] 定义 `manifest.yaml` schema（Python dataclass 或 pydantic）
- [ ] 修改角色加载逻辑，先读取 `manifest.yaml`，再加载 `personality.yaml`
- [ ] 向后兼容：无 `manifest.yaml` 时降级为旧格式加载
- [ ] 写 `tools/validate_character.py`：验证角色包格式是否符合规范

### 6.2 用户数据根目录完善

> `KOKORO_DATA_DIR` 和基础路径已在 Phase 0 建立（见 `src/config.py`）。Phase 6 补充角色包子目录结构。

目标完整目录结构：

```
$KOKORO_DATA_DIR/            # 默认 ./data，可通过环境变量自定义
├── characters/              # 已安装的角色包（Phase 6 从代码仓库 characters/ 迁入）
├── memories/                # 各角色的记忆数据（Phase 2 已落盘）
│   └── <character_id>/
│       ├── summaries.jsonl
│       ├── facts.json
│       └── chroma/          # 向量数据库（Phase 3 之后）
├── settings.yaml            # 全局设置（Phase 6 新增）
└── logs/                    # 会话日志（Phase 0 已落盘）
```

任务：

- [ ] 确认所有路径均通过 `src/config.py` 计算，无硬编码绝对路径（Phase 0 应已完成）
- [ ] 增加 `settings.yaml` 全局配置读写
- [ ] 不写注册表（Windows）/ 不写 `/etc`（Linux/Mac）

### 6.3 导入导出命令

```bash
# 备份整个用户目录
python main.py export --output backup.zip

# 恢复
python main.py import backup.zip

# 仅迁移某个角色（含记忆）
python main.py export --character asuka --output asuka_full.zip
```

任务：

- [ ] 实现 `export` 命令：将 `characters/` + `memories/<id>/` + `logs/` 打包为 zip
- [ ] 实现 `import` 命令：解压到当前数据根目录，冲突时提示用户
- [ ] 单角色导出：只打包指定角色的目录和记忆
- [ ] 写集成测试：导出后导入，验证角色和记忆数据完整

### 6.4 角色包安装（本地工坊）

```bash
# 安装本地角色包
python main.py install ./asuka.charpkg

# .charpkg 实际是 zip，解压到 characters/ 目录下
```

任务：

- [ ] 实现 `.charpkg` 格式（zip 封装，包含 manifest.yaml 校验）
- [ ] 实现 `install` 命令：解压、校验 manifest、写入 `characters/<id>/`
- [ ] 安装前校验目标目录不存在路径穿越、覆盖系统文件等安全问题
- [ ] 支持拖入 Tauri 窗口安装（Phase 6 与 UI 层联动）

### 6.5 插件接口设计

目标：触发器、记忆处理器、TTS 后端均设计为可插拔接口。

```python
# 触发器插件示例
class CustomTrigger(BaseTrigger):
    def check(self, context: PerceptionContext) -> bool: ...
    def get_event_tag(self) -> str: ...

# 记忆处理器插件示例
class CustomMemoryProcessor(BaseMemoryProcessor):
    def extract_facts(self, conversation: list) -> list[dict]: ...

# TTS 后端插件示例（已在能力层定义 BaseTTSBackend）
```

任务：

- [ ] 确认 `BaseTrigger`、`BaseTTSBackend` 等基类已定义（Phase 3 / Phase 5 应已完成）
- [ ] 定义 `BaseMemoryProcessor` 接口
- [ ] 首版插件发现机制优先支持受信任本地插件目录和显式启用配置，不默认执行任意第三方 Python 包
- [ ] 若扫描插件目录，统一通过 manifest 注册，运行前先做版本与接口兼容检查
- [ ] 写插件开发文档

### 验收

- 换机器后，`import backup.zip` 可完整恢复角色、记忆和日志
- 新增角色只需新建 `characters/<name>/` 目录，无需改 `src/` 代码
- 角色包安装和卸载不影响其他角色数据
- 插件接口有清晰文档，第三方可实现自定义触发器
