## Phase 0–1：目录结构先行约定

虽然扩展机制是后置的，但目录结构从一开始就应规范，避免 Phase 6 重构成本过高。

### 任务清单

- [ ] 确认角色目录结构为 `characters/<name>/personality.yaml`，不改动 `src/`
- [ ] `logs/` 按日期或会话 ID 命名，不混入角色数据
- [ ] 记忆数据存放路径提前规划（Phase 2 实现时直接按规范落盘）：`memories/<character_id>/`

---

## Phase 2：记忆路径规范落盘

在实现记忆层时，直接按迁移友好的路径落盘。

### 任务清单

- [ ] 摘要记忆：`memories/<character_id>/summaries.jsonl`
- [ ] 长期记忆：`memories/<character_id>/facts.json`
- [ ] 路径通过配置变量管理，不硬编码
- [ ] 确认所有路径为相对路径或可配置根目录，支持未来迁移

## Phase 6：扩展与迁移完整实现

### 6.1 角色包格式标准化

目标角色目录结构：

```
characters/
└── asuka/
    ├── manifest.yaml        # 包元数据
    ├── personality.yaml     # 人格配置
    ├── live2d/              # Live2D 模型文件（可选）
    │   ├── asuka.model3.json
    │   └── ...
    ├── voice/               # 语音素材（可选，用于 TTS 克隆）
    │   └── samples/
    ├── corpus/              # 台词库（可选，用于 RAG）
    │   └── lines.jsonl
    └── preview.png          # 预览图（可选）
```

`manifest.yaml` 结构：

```yaml
id: "asuka-langley-tv"
name: "惣流·明日香·兰格雷"
version: "1.0.0"
author: ""
description: ""
license: "CC BY-NC-SA"
requires:
  platform: ">=0.1.0"
  live2d_model: true
  tts_voice: true
```

任务：

- [ ] 定义 `manifest.yaml` schema（Python dataclass 或 pydantic）
- [ ] 修改角色加载逻辑，先读取 `manifest.yaml`，再加载 `personality.yaml`
- [ ] 向后兼容：无 `manifest.yaml` 时降级为旧格式加载
- [ ] 写 `tools/validate_character.py`：验证角色包格式是否符合规范

### 6.2 用户数据根目录统一

目标目录结构：

```
~/ai-agent-platform/         # 可通过环境变量 KOKORO_DATA_DIR 自定义
├── characters/              # 已安装的角色包
├── memories/                # 各角色的记忆数据
│   └── <character_id>/
│       ├── summaries.jsonl
│       ├── facts.json
│       └── chroma/          # 向量数据库（Phase 3 之后）
├── settings.yaml            # 全局设置
└── logs/
```

任务：

- [ ] 引入 `KOKORO_DATA_DIR` 环境变量，默认值为 `~/ai-agent-platform/`
- [ ] 所有文件读写路径统一基于此根目录计算
- [ ] 应用内无硬编码绝对路径
- [ ] 不写注册表（Windows）/ 不写 `/etc`（Linux/Mac）

### 6.3 导入导出命令

```bash
# 备份整个用户目录
python main.py export --output backup.zip

# 恢复
python main.py import backup.zip

# 仅迁移某个角色（含记忆）
python main.py export --character asuka --output asuka_full.zip
```

任务：

- [ ] 实现 `export` 命令：将 `characters/` + `memories/<id>/` + `logs/` 打包为 zip
- [ ] 实现 `import` 命令：解压到当前数据根目录，冲突时提示用户
- [ ] 单角色导出：只打包指定角色的目录和记忆
- [ ] 写集成测试：导出后导入，验证角色和记忆数据完整

### 6.4 角色包安装（本地工坊）

```bash
# 安装本地角色包
python main.py install ./asuka.charpkg

# .charpkg 实际是 zip，解压到 characters/ 目录下
```

任务：

- [ ] 实现 `.charpkg` 格式（zip 封装，包含 manifest.yaml 校验）
- [ ] 实现 `install` 命令：解压、校验 manifest、写入 `characters/<id>/`
- [ ] 支持拖入 Tauri 窗口安装（Phase 6 与 UI 层联动）

### 6.5 插件接口设计

目标：触发器、记忆处理器、TTS 后端均设计为可插拔接口。

```python
# 触发器插件示例
class CustomTrigger(BaseTrigger):
    def check(self, context: PerceptionContext) -> bool: ...
    def get_event_tag(self) -> str: ...

# 记忆处理器插件示例
class CustomMemoryProcessor(BaseMemoryProcessor):
    def extract_facts(self, conversation: list) -> list[dict]: ...

# TTS 后端插件示例（已在能力层定义 BaseTTSBackend）
```

任务：

- [ ] 确认 `BaseTrigger`、`BaseTTSBackend` 等基类已定义（Phase 3 / Phase 5 应已完成）
- [ ] 定义 `BaseMemoryProcessor` 接口
- [ ] 实现插件发现机制：扫描 `platform/plugins/` 目录，自动注册插件
- [ ] 写插件开发文档

### 验收

- 换机器后，`import backup.zip` 可完整恢复角色、记忆和日志
- 新增角色只需新建 `characters/<name>/` 目录，无需改 `src/` 代码
- 角色包安装和卸载不影响其他角色数据
- 插件接口有清晰文档，第三方可实现自定义触发器
