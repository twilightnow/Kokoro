# 记忆层

## 目标
定义 Kokoro 当前三级记忆的真实实现: 工作记忆、摘要记忆、长期记录仓库，以及它们如何汇总成 `MemoryContext`，同时明确未来向量检索迁移的接入边界。

## 范围
### 包含
- `WorkingMemory`、`SummaryMemory`、`LongTermMemory`、`MemoryService` 的当前行为。
- `MemoryRecord`、候选/确认生命周期、`retrieve()` 与 `build_context()` 的当前规则。
- `MemoryContext` 中 `summary_items`、`long_term_items`、`preference_items`、`boundary_items`、`event_items` 的真实来源。
- 增量结算游标、空闲/轮数/shutdown 触发、候选冲突、摘要裁剪、token 预算裁剪、`memory_ops.jsonl` 和 admin 管理接口。

### 不包含
- 不把 `PromiseMemory`、提醒调度或向量数据库写成当前运行时依赖。
- 不定义 LLM provider 的底层实现。
- 不把工作记忆 token 上限或摘要压缩写成已实现。

## 文档同步边界
- 本文以当前 `WorkingMemory`、`SummaryMemory`、`LongTermMemory`、`MemoryService` 和 admin 路由行为为准；`PromiseMemory`、提醒调度和向量数据库均不属于当前记忆层运行时依赖。
- 系统总览只描述记忆仓库在对话链路中的位置、增量结算触发点和未来向量检索边界；事实生命周期、候选屏蔽和文件格式以本文为准。
- IPC 协议只维护 `/admin/memories/*` 与 `/admin/debug/flush-session` 的请求/响应契约；候选确认、冲突处理和 prompt 注入规则以本文为准。
- 管理界面只消费记录仓库与兼容候选字段；Memories 页不得绕过 `facts.json` 的记录仓库语义直接管理向量召回结果。
- 人格层只消费 `MemoryContext` 的结构化结果；`MemoryContext` 字段未变化时，不需要同步修改人格层文档。
- 长期伴侣运行时仅引用记忆约束和结算边界；提醒与主动陪伴数据独立于当前记忆仓库。

## 流程
1. `WorkingMemory` 维护最近 `max_rounds=10` 轮对话；若下一轮发送前将发生截断，`ConversationService` 先结算尚未落盘的新增片段，再执行 `truncate()`。
2. `ConversationService._prepare_turn()` 追加当前用户输入到工作记忆，然后调用 `MemoryService.get_context()`。
3. `get_context()` 先通过 `retrieve()` 读取 `facts.json` 中的活动记录，只向 prompt 注入 `confirmed` 状态的记录；若同一非 `event` key 存在待确认 `candidate`，则旧 `confirmed` 记录也会暂时从 prompt 中屏蔽。
4. `get_context()` 再读取最近摘要，按新到旧尝试填充剩余预算，并把选中的摘要恢复为旧到新顺序。
5. `build_context()` 按 `boundary`、`preference`、`fact`、`event` 优先级与 token 预算挑选条目；无重复时 `event_items` 继续使用原始 key，有重复时才追加记录后缀避免覆盖。
6. 超出预算的事实或摘要会写入 `truncation.log`。
7. `_finalize_turn()` 在回复写入工作记忆后递增结算轮数计数；达到 `KOKORO_MEMORY_SETTLE_TURNS` 时立即结算，否则重置空闲计时器。
8. 空闲计时器超过 `KOKORO_MEMORY_IDLE_SECONDS` 时，`ConversationService._handle_idle_timeout()` 自动结算当前未落盘片段。
9. CLI 退出、角色切换、FastAPI lifespan shutdown、桌面端退出前的 `/admin/debug/flush-session` 都会触发同一条结算路径。
10. `ConversationService` 通过 `last_persisted_message_count` 只把上次成功结算后新增的消息切片传给 `MemoryService.on_session_end()`，避免重复摘要和重复抽事实。
11. `on_session_end()` 总是尝试生成最多 100 字的摘要；只有新增用户消息命中显式事实/偏好/边界/近期事件特征时，才追加一次事实抽取调用。
12. 自动事实抽取默认写入 `candidate` 记录，不直接改写 `confirmed`；`memory_ops.jsonl` 会记录本次抽取输入类别和写入动作。
13. 同一事实 key 写入不同 value 或 category 时，长期记忆保留旧值并新增一个 `candidate` 记录；手工确认或显式写入 `confirmed` 时，旧记录会被归档或被新记录取代。
14. admin 通过 `/admin/memories/*` 浏览、修改、解决候选、删除和导出事实与摘要；当前接口继续返回 `pending_confirm`、`pending_value` 等兼容字段，但底层实际操作的是记录仓库。
15. AIRI 合并修正(追加:2026-04-28): `retrieve()` 选中 `confirmed` 记录进入 `MemoryContext` 后，记忆仓库必须对这些记录执行 touch，更新 `last_accessed` 和 `access_count`；touch 失败不得阻断本轮聊天。
16. AIRI 合并修正(追加:2026-04-28): prompt 注入长期记忆时必须携带 category、confidence 或 status 来源摘要、importance 和 source 摘要；若字段缺失，输出 `[待定:旧记忆缺少来源字段]` 而不是编造来源。

## 约束与规则
- 工作记忆只保存在进程内，不落盘。
- 摘要最大长度 100 字；空摘要不写入。
- 长期记忆允许的 category 只有 `fact`、`preference`、`boundary`、`event`。
- 长期记忆状态只有 `candidate`、`confirmed`、`rejected`、`archived`；只有 `confirmed` 可进入 prompt。
- `facts.json` 是当前唯一权威存储；未来向量数据库只能作为检索索引或召回层，不能绕过记录仓库直接作为事实真源。
- 若未来引入向量检索，向量条目应从记录仓库投影生成，并保留 `record_id` 回指权威记录。
- 当前没有 `PromiseMemory`，也没有提醒执行器。
- AIRI 合并修正(追加:2026-04-28): 长期记忆类型固定为 `fact`、`preference`、`boundary`、`event`；新增类型必须先更新本文件、API 文档和管理界面文档。
- AIRI 合并修正(追加:2026-04-28): pgvector、embedding 和大规模消息 RAG 不属于当前 P0/P1 范围；只有当 JSON 记录仓库无法满足检索性能或召回质量时，才允许把向量库作为索引层接入。
- AIRI 合并修正(追加:2026-04-28): 管理界面必须支持确认、编辑、删除、归档候选记忆；删除或归档后的记录不得进入 prompt，也不得被向量索引重新召回为事实真源。
- 事实冲突不会自动覆盖旧值，必须由后续逻辑或 admin 明确处理；存在待确认候选时，对应旧值默认不再注入 prompt。
- 默认一次结算最多包含 1 次摘要调用和 0 或 1 次事实抽取调用；闲聊切片只做摘要，不做事实抽取。
- `MemoryContext` 只暴露结构化结果，不暴露底层文件格式给人格层。

## 验收标准
- 连续发送超过 10 轮对话；工作记忆保持最近 10 轮的消息数量上限。
- 连续对话达到 `KOKORO_MEMORY_SETTLE_TURNS`；新增消息片段被自动写入摘要，且 shutdown 时不会重复写同一片段。
- 单轮对话后触发空闲超时；新增消息片段被自动结算一次，再次触发空闲回调不会重复写入。
- 自动抽取 `{"user_name":"小明"}`；`facts.json` 新增 `candidate` 记录，`get_confirmed_facts()` 不会直接出现该值。
- 写入同一 `key` 的不同 `value`；旧 `confirmed` 记录保留，新增 `candidate` 记录，且 `get_context()` 不再把旧值继续注入 prompt。
- 触发 CLI 退出、角色切换、FastAPI shutdown 或 `/admin/debug/flush-session`；`summaries.jsonl` 新增一条摘要或在 LLM 失败时保持文件可读。
- 结算切片只包含“哈哈”“继续聊”等闲聊；摘要调用执行一次，事实抽取不会额外调用。
- 调用 `get_context()` 且预算不足；被裁掉的事实或摘要写入 `truncation.log`。
- 调用 `/admin/memories/<id>/facts/{key}/resolve` 且 `adopt_new=true`；候选记录升级为 `confirmed`，被替代的旧记录转入 `archived`。
- 调用 `get_context()` 召回 2 条 confirmed 记忆；对应记录的 `access_count` 增加且 `last_accessed` 更新为本次调用时间。
- 在 admin 将一条 confirmed 记忆归档；再次调用 `get_context()`，该记忆不出现在 `MemoryContext` 和 prompt 中。
