# 长期伴侣运行时

长期伴侣运行时负责把 Kokoro 从“用户输入一轮、角色回复一轮”的对话系统，扩展成“能持续存在、记住关系、管理承诺、适度主动”的桌面伴侣系统。

这份文档只描述设计边界，不定义阶段任务。

## 设计目标

- 让关系状态成为独立运行时状态，而不是普通记忆文本
- 让主动行为由调度器统一决策，而不是触发器命中后立刻说话
- 让提醒、约定和待跟进事项结构化保存
- 让生活事件、偏好和边界进入可治理记忆
- 让情绪具备连续性，而不是固定轮数标签

## 核心模块

### CompanionRuntime

长期运行时的总协调器。

职责：

- 周期性检查感知上下文、提醒、关系状态和情绪恢复
- 调用主动调度器判断是否需要介入
- 将运行时事件写入日志或事件记忆
- 向 UI 推送轻量状态变化，例如表情、注意力、气泡和提醒

不负责：

- 直接生成角色台词
- 直接修改角色人格配置
- 直接读取底层记忆文件

### RelationshipService

关系状态服务。

职责：

- 读取和保存 `RelationshipState`
- 根据互动事件缓慢更新关系状态
- 生成可注入 prompt 的关系摘要
- 提供管理界面查看和重置能力

关系状态不应由 LLM 直接决定。LLM 可以提出“关系事件候选”，最终数值变化由规则层控制。

### ProactiveScheduler

主动行为调度器。

职责：

- 接收感知事件、提醒事件、关系事件和情绪事件
- 判断是否介入
- 判断介入等级
- 应用冷却、勿扰、每日上限和用户偏好
- 输出结构化 `ProactiveDecision`

主动行为等级：

- `silent`：保持沉默，只记录状态
- `expression`：只改变表情、动作或注意力
- `short_message`：一句短提醒或问候
- `conversation`：完整对话式介入

触发器只负责发现事件，调度器负责决定是否真的打扰用户。

### RoutineReminderService

承诺和提醒服务。

职责：

- 保存一次性提醒
- 保存周期性提醒
- 保存角色答应过的待跟进事项
- 标记完成、跳过、过期和取消
- 将到期提醒转成主动调度事件

提醒不应只存在于自然语言记忆里。凡是有时间、重复规则或完成状态的内容，都应结构化保存。

### EmotionTimeline

情绪连续状态。

职责：

- 保存当前主要情绪
- 保存强度、原因、来源和开始时间
- 根据时间、互动和事件自然恢复
- 为 prompt、TTS、Live2D、3D、主动调度提供一致状态

情绪可以影响表达方式，但不应越权改变关系状态和安全策略。

## 运行时状态

建议按角色和用户本地数据隔离保存：

```text
data/
  runtime/
    <character_id>/
      relationship.json
      emotion_timeline.json
      reminders.json
      events.jsonl
      preferences.json
      boundaries.json
```

实际路径可以演进，但设计上应保持：

- 角色包资源和用户运行时数据分离
- 运行时数据可迁移
- 每类状态有独立 schema_version
- 损坏时能局部降级，不影响主对话

## RelationshipState 设计

关系状态应保持少量、可解释、可回滚。

建议字段：

- `intimacy`：亲密度
- `trust`：信任度
- `familiarity`：熟悉度
- `interaction_quality_recent`：近期互动质量
- `preferred_addressing`：称呼偏好
- `relationship_type`：朋友、恋人、家人、搭档等关系类型
- `boundaries_summary`：边界摘要
- `dependency_risk`：依赖风险
- `updated_at`
- `change_reasons`

设计约束：

- 数值变化必须慢
- 必须记录变化原因
- 管理界面必须可查看和重置
- 不应把关系状态写回角色包

## EventMemory 设计

生活事件用于保存“共同经历”和“近期状态”，不同于长期事实。

事件类型示例：

- `shared_moment`：共同经历
- `user_goal`：近期目标
- `stress_source`：压力源
- `anniversary`：重要日期
- `promise`：承诺相关事件
- `routine`：习惯或作息事件

事件记忆应支持：

- 时间戳
- 重要性
- 是否可主动提及
- 是否敏感
- 来源
- 过期策略

## 主动调度输入

主动调度器不直接读取所有模块内部状态，而接收标准事件：

- `PerceptionEvent`：深夜、久坐、空闲、频繁切窗口、游戏等
- `ReminderEvent`：提醒到期、周期检查、承诺跟进
- `RelationshipEvent`：久未互动、互动质量变化、关系阶段变化
- `EmotionEvent`：情绪恢复、情绪过强、用户安抚成功
- `SystemEvent`：LLM 不可用、TTS 不可用、资源降级

统一事件可以避免主动行为逻辑散落在感知、记忆和 UI 层。

## Prompt 注入原则

长期伴侣状态不能把完整 JSON 直接塞进 prompt。应先生成短摘要：

- 当前关系摘要
- 最近重要事件
- 用户偏好和边界
- 当前提醒上下文
- 当前情绪原因和强度

摘要必须受 token budget 控制，且敏感边界优先级高于普通事件。

## UI 表现关系

运行时状态应同时驱动文字和表现层：

- 关系状态影响称呼和主动频率
- 情绪强度影响表情、动作和语音
- 主动调度结果影响是否显示气泡
- 提醒事件影响快捷按钮
- 勿扰状态影响是否只做表情响应

UI 只消费运行时输出，不应自行判断“该不该主动开口”。

## 设计边界

- 伴侣感来自连续状态，不来自单次夸张回复
- 主动性必须节制，沉默也是一种有效输出
- 关系成长必须可解释、可重置
- 任何“答应过的事”都应结构化保存
- 安全边界优先级高于角色设定和关系状态
- 长期状态不能破坏角色核心人格
