# 人格层

人格层是当前项目的核心。它的职责不是“生成任意好看的 prompt”，而是把角色配置、情绪状态、记忆和感知收敛成一个稳定的 system prompt。

## 当前模块

- `character.py`：角色配置数据结构
- `loader.py`：YAML 加载与校验
- `emotion.py`：情绪事件检测和状态衰减
- `prompt_builder.py`：统一拼装 system prompt

## 当前角色配置

角色文件位于：

- `characters/<id>/personality.yaml`

当前真正被代码消费的字段有：

- `name`
- `version`
- `schema_version`
- `personality.core_fear`
- `personality.surface_trait`
- `personality.hidden_trait`
- `behavior_rules`
- `proactive_style.idle_too_long`
- `proactive_style.user_working_late`
- `proactive_style.user_gaming`
- `forbidden_words`
- `verbal_habits`
- `mood_expressions`
- `emotion_triggers`

保持约束：

- YAML 只描述角色，不写业务流程
- 字段尽量固定，不做无限扩张
- 角色切换只换配置，不改业务代码

## 当前情绪模型

当前实现是一个极简状态机：

- 状态：`normal` / `happy` / `angry` / `shy` / `cold`
- 附加值：`persist_count`

工作方式：

1. 先用 `detect_event()` 根据关键词匹配情绪事件
2. 命中则切到对应情绪并重置持续轮数
3. 未命中则衰减，最终回到 `normal`

这里故意很轻，不引入 affection、energy、关系值等复杂状态。

## 当前 prompt 组装

人格层不让其他模块直接拼 prompt，统一通过：

- `PromptContext`
- `build_system_prompt()`

`PromptContext` 当前包含：

- `character`
- `emotion`
- `memory`
- `perception`

这样做的意义是：

- 记忆层只负责给结构化记忆
- 感知层只负责给结构化场景
- 最终表达方式仍由人格层统一决定

## 当前 system prompt 内容

当前 prompt 主要由这些部分组成：

- 角色身份
- 核心人格三段
- 行为规则
- 口癖
- 禁用词
- 当前情绪描述
- 背景记忆
- 当前场景
- 回复约束

其中记忆和感知都是可选注入，不会反向依赖人格层内部实现。

## 当前没有做的事

下面这些内容目前还没有落地，不应写成既成事实：

- 台词库 RAG
- embedding 检索
- 关系状态系统
- 更复杂的人格数值系统

如果后面需要引入，建议仍然作为人格层的输入，而不是绕过人格层直接干预回复。

## 保持简单的边界

- 应用层不直接拼 system prompt
- API 层不直接读角色 YAML
- LLM client 不关心角色细节
- 新的人格信息优先加到 `PromptContext`，不要跨层透传散乱字段

人格层要继续保持“统一出口”，这是当前架构里最值钱的解耦点。
