## 表现形式

立绘渲染方式从轻到重，可按开发阶段逐步升级，人格层与表现层解耦，切换渲染方案不影响对话逻辑：

| 形式 | 说明 | 适合阶段 |
|------|------|---------|
| 纯文字气泡 | 无立绘，仅对话气泡 | Demo / Phase 1 占位 |
| 静态立绘 + 表情差分 | PNG 切换，情绪对应不同表情图 | Phase 2 过渡 |
| 漫画风格帧动画 | 关键动作用少量帧表现，风格化强，轻量 | Phase 2–3 |
| Live2D 骨骼动画 | 流畅自然，当前主选方案 | Phase 3 |
| 3D 模型（VRM） | 自由度最高，备选 | Phase 3 后 |

**漫画 / 动画展现方向补充：**

漫画风格适合在性能受限或首版快速验证阶段替代 Live2D。核心思路是用少量差分图（正常 / 生气 / 害羞 / 冷漠）加简单帧动画（如眨眼、说话口型）模拟表现力，比纯静态图更有生命感，比 Live2D 实现成本低得多。

动画气泡也可以配合漫画风格：说话时气泡以弹出动画出现，消失时淡出，而不是像聊天记录堆叠。这与漫画对白框的语言一致，比 IM 气泡更贴合角色陪伴的场景感。

## 功能设计

- 桌面悬浮窗，默认右下角，可拖动
- 系统托盘常驻，右键菜单支持角色切换
- 立绘根据情绪状态切换表情和动作
- 对话气泡显示在立绘旁边，不遮挡屏幕主要区域

首版目标是“低打扰、常驻稳定、交互足够轻”，不是一开始追求复杂表现力。

UI 的定位不是“再做一个聊天窗口”，而是让角色以低成本存在于桌面上。用户平时感知到她，主要依赖状态、气泡和轻交互，而不是长时间展开的大面板。

**窗口模式：**

| 模式 | 说明 |
|------|------|
| 对话模式 | 显示输入框，支持打字 / 语音输入 |
| 锁定模式 | 纯角色展示，隐藏输入框，仅响应主动介入和轻量交互 |

建议进一步理解为两类使用方式：

- **常驻陪伴**：默认锁定模式，强调存在感，不强调输入效率
- **主动聊天**：用户点击后进入对话模式，才展开完整对话

**交互方式（从轻到重）：**

- 点击立绘 → 触发随机互动（最轻，无需输入）
- 快捷回应按钮 → 预设短回复（"嗯" / "知道了" / "说下去"），一键发送
- 打字输入 → 完整对话
- 语音输入 → 说话即对话

锁定模式下保留点击和快捷回应，适合作为桌面挂件长期运行。

推荐交互优先级：

1. 用户先看到她“在场”
2. 用户再接收一句短气泡或一个表情变化
3. 用户如果愿意，再点开完整聊天

也就是说，首版主入口应该是挂件和短反馈，完整聊天面板是次入口。

首版实现建议收敛到：

- 托盘常驻
- 对话气泡
- 输入框
- 基础表情切换

复杂动作同步、语音输入、VTuber 扩展都后置。

补充建议：

- 默认回复尽量短，避免桌面场景下出现长段文本
- 主动介入以一句话加快捷回应为主，不默认展开多轮对话
- 让”点开聊天”成为用户主动选择，而不是系统默认状态

## 气泡范式

气泡设计不走聊天记录式（消息列表 + 滚动），而走**漫画对白框**式：

- 每次只显示当前这句话，说完后自动淡出
- 没有历史堆叠，没有滚动条
- 历史记录若需要查看，通过托盘菜单单独打开，不是默认展示
- 气泡出现 / 消失有轻微动画，不是瞬切

这与立绘陪伴的场景感一致，避免把体验做成”又一个聊天软件”。

## 窗口行为

**点击穿透：**

立绘本体可点击交互，立绘以外的透明区域穿透到底层窗口。用户在透明区域的操作直接落到底层应用，不被挡住。Tauri 支持分区点击穿透。

**默认位置与边缘吸附：**

- 默认位置改为屏幕右侧边缘居中，避开任务栏和系统通知区域
- 拖动到屏幕边缘时自动吸附并露出立绘一半，不完全隐藏
- 吸附状态下点击或气泡出现时自动滑出

**点击与拖动区分：**

- 短按（< 300ms）→ 触发轻交互
- 长按拖动 → 移动位置
- 双击 → 切换对话模式 / 锁定模式

## 主动介入的出现方式

主动介入不应直接弹出气泡，应有前置提示让用户感知到她”想说话”：

1. 立绘先有轻微动作（如抬头、眨眼加速）
2. 停顿约 1 秒
3. 气泡以弹出动画出现，附带一句短话 + 快捷回应按钮
4. 若用户无响应，N 秒后气泡淡出，不强迫回应

这样主动介入是”她在引起注意”，而不是”系统在打扰用户”。

---

## 输入框与响应节奏

**输入框出现时机：**

- 默认不显示输入框
- 用户点击立绘或双击进入对话模式后，输入框淡入
- 她回复完毕后，输入框保持 N 秒等待，无操作则淡出，回到常驻状态

**响应节奏：**

- 她回复前显示”思考指示”（如省略号动画或轻微动作），避免立即输出显得机械
- 回复以流式输出呈现，逐字或逐词显示，比一次性弹出更像在说话
- 回复内容应尽量短，一句话为主，必要时才有第二句
