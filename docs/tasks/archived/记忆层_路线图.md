---

## 模块职责

- **负责**：对话历史管理、会话摘要生成与存储、长期记忆读写
- **不负责**：决定记忆内容如何影响角色输出（由人格层决定）
- **约束**：记忆只向人格层提供输入，按角色隔离存储

### 编码边界

- 记忆层负责"存什么、怎么读写、怎么裁剪"，不负责"最终怎么写进 prompt"。
- 记忆层对外优先暴露 `MemoryService` 和 `MemoryContext`，不要让上层直接碰文件结构或路径。
- 所有文件路径基于 `KOKORO_DATA_DIR` 环境变量计算（见总览 §6 Phase 0），不硬编码。
- 首版优先文件存储，但文件格式和路径计算应封装在单一模块，避免路径字符串散落到业务代码。

### MemoryContext 结构（Phase 0 起固定，Phase 2 实现内容）

```python
# src/memory/context.py
@dataclass
class MemoryContext:
    summary_items: list[str] = field(default_factory=list)
    # 最近 N 条摘要，每条 ≤100 字，由摘要记忆层填充

    long_term_items: dict[str, str] = field(default_factory=dict)
    # key→value 已确认事实，如 {"user_name": "小明"}
    # 由长期记忆层填充，注入时总长度不超过 budget
```

Phase 0 直接用空的 `MemoryContext()` 占位，Phase 2 再填充内容，**不改接口**。

## 三级记忆结构

```
长期记忆
├── 用户基本信息（名字、职业、习惯）
└── 关键关系节点（第一次对话、重要事件）

摘要记忆
├── 历史对话摘要（每次会话结束后生成）
└── 承诺/约定记录

工作记忆
└── 最近 10 轮对话（运行时维护，不落盘）
```

## Phase 0：工作记忆收口

目标：确保工作记忆在主流程中正确截断，不让 prompt 无限增长。

### 任务清单

- [ ] 确认 `working_memory.py` 的截断逻辑（`max_rounds=10`）在每轮发送前执行

## Phase 2：轻量记忆落地

### 2.1 会话摘要记忆

目标：会话结束后生成摘要，作为下次会话的上下文背景。

技术方案：

- 会话结束时调用 LLM 生成摘要（50-100 字），写入 JSONL 文件
- 摘要按角色隔离存储：`memories/<character_id>/summaries.jsonl`
- 下次会话开始时，加载最近 N 条摘要，注入 system prompt 的摘要区域

任务：

- [ ] 在应用编排层的会话结束钩子处调用摘要生成，不把摘要逻辑直接塞进 `main.py`
- [ ] 实现 `src/memory/summary_memory.py`：`generate_summary(history) -> str`，`save_summary(character_id, summary)`，`load_recent_summaries(character_id, n=3) -> list[str]`
- [ ] 由记忆层产出 `MemoryContext.summary_items`，人格层只消费结果，不直接读取 `summaries.jsonl`
- [ ] 写单元测试：给定对话历史，验证摘要生成不为空且长度合理

### 2.2 最小长期记忆

目标：只记录用户明确说出的事实，不乱记。

存储方案（首版用 JSON 文件，不依赖 ChromaDB）：

```json
// memories/<character_id>/facts.json
{
  "user_name": "值",
  "user_occupation": "值",
  "preferences": [],
  "agreements": []
}
```

写入原则：

- 用户明确说出的事实，才进入长期记忆
- 模型推断出的内容，默认只进入摘要记忆
- 发生冲突时保留旧记录并标记 `pending_confirm: true`，不直接覆盖

任务：

- [ ] 实现 `src/memory/long_term_memory.py`：`read_facts(character_id)`，`write_fact(character_id, key, value)`，`flag_conflict(character_id, key, new_value)`
- [ ] 定义 `FactRecord` 结构，至少包含 `key`、`value`、`source`、`updated_at`、`pending_confirm`
- [ ] 由记忆层负责将长期记忆裁剪成精简 `MemoryContext.long_term_items`，人格层只做展示性注入
- [ ] 写单元测试：连续写入同一 key 两次不同值，验证冲突标记而非覆盖
- [ ] 确保每个角色的记忆存储路径隔离，不同角色不共享 facts.json

### 2.3 记忆注入长度控制

目标：记忆内容注入 prompt 时不能无限增长。

任务：

- [ ] 摘要记忆：只注入最近 3 条，每条不超过 100 字
- [ ] 长期记忆：只注入高权重事实，总计不超过 200 tokens
- [ ] 在记忆层增加 `build_memory_context(character_id, token_budget: int = 200) -> MemoryContext` 接口，先裁剪后返回；`token_budget` 指允许注入的最大 token 数（粗估按 4 字符/token 换算），避免人格层反向理解记忆存储细节
- [ ] 记录哪些记忆因为预算被截断，便于复盘调参

### 验收

- 下次会话开始时，prompt 中包含上次摘要
- 用户说"我叫 XX"后，facts.json 中写入用户名称
- 角色切换后，各自使用独立的 summaries.jsonl 和 facts.json
- 记忆注入后 prompt 总长度不超过设定阈值

## 不建议在此阶段做的事

- Phase 2 之前引入 ChromaDB（JSON 文件足够，不需要向量数据库）
- 自动从模型输出中提取记忆（容易乱记，先用明确事实写入）
- `mem0` 在验证写入原则之前引入
- 在 `prompt_builder.py` 或 UI 层里直接打开 `facts.json` / `summaries.jsonl`
