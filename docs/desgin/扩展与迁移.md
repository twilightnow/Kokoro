# 扩展与迁移

扩展与迁移设计的核心目标是：角色包、用户数据、运行时状态和程序代码彼此分离。这样角色可以安装、升级、卸载，用户记忆和关系状态也能被保留、导出或清空。

## 目录边界

建议保持三类数据分离：

```text
characters/             # 角色包和静态资源
data/memories/          # 用户记忆
data/runtime/           # 关系、提醒、情绪等运行时状态
data/logs/              # 会话和诊断日志
```

设计原则：

- 角色包是内容，不是用户状态
- 用户数据不写回角色包
- 运行时状态不写入 `personality.yaml`
- 删除角色包不默认删除用户数据
- 清空用户数据不删除角色资源

## 角色包结构

推荐角色包结构：

```text
characters/<character_id>/
  manifest.yaml
  personality.yaml
  assets/
    live2d/
    model3d/
    sprites/
  voices/
  prompts/
```

### personality.yaml

只描述角色人格和表达规则。

不应包含：

- 用户记忆
- 当前关系状态
- API key
- 本机绝对路径
- 当前会话状态

### manifest.yaml

描述角色包元数据和资源能力。

建议包含：

- `id`
- `display_name`
- `version`
- `schema_version`
- `author`
- `display`
- `voice`
- `capabilities`
- `asset_paths`

manifest 不应承载人格规则。

## character_id

`character_id` 是所有跨模块存储和索引的稳定主键。

设计要求：

- 使用稳定 ASCII ID
- 不使用显示名做主键
- 升级角色包时保持不变
- 记忆、日志、运行时状态都按 `character_id` 隔离

## 安装与升级

角色安装流程应是原子化的：

1. 解包到临时目录
2. 校验 manifest
3. 校验 personality
4. 校验资源路径
5. 检查 `character_id` 冲突
6. 写入正式目录

角色升级流程：

- 同 `character_id` 允许覆盖角色包
- 保留用户数据
- 失败时回滚角色包
- schema 变化必须走迁移

## 卸载与删除

卸载分两种动作：

- 卸载角色包：删除 `characters/<id>`
- 删除角色数据：删除 `data/memories/<id>`、`data/runtime/<id>` 和相关日志

这两个动作必须在 UI 和 API 上分开，不能混用。

## Schema 迁移

需要迁移的对象：

- 角色包 manifest
- personality
- 记忆数据
- 运行时状态
- 日志格式

设计原则：

- 每类数据有独立 `schema_version`
- 迁移应可重复执行或能识别已迁移状态
- 迁移失败不应破坏原始数据
- 重大迁移前应备份

## 导入导出

导出类型：

- 角色包导出：只包含静态角色资源
- 用户数据导出：包含记忆、关系、提醒、偏好和边界
- 调试导出：包含脱敏日志和配置摘要

导出时必须避免：

- API key
- 未脱敏敏感窗口标题
- 用户未选择导出的隐私数据

## 插件边界

如果未来引入插件，插件不应直接改核心状态文件。

插件只能通过公开 API：

- 注册能力
- 读取允许的上下文
- 提供 UI 面板
- 写入限定数据类型

插件不能：

- 读取 provider key
- 绕过安全策略
- 直接修改关系状态
- 直接拼接 prompt

## 设计约束

- 内容包和用户数据分离
- 升级角色不丢记忆
- 删除角色不误删用户数据
- schema 演进必须可迁移
- 导出默认脱敏
- 扩展能力不能绕过安全边界
