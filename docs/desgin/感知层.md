# 感知层

当前感知层已经落地，但仍然严格控制在轻量、本地、低打扰的范围里。

## 当前模块

- `input_tracker.py`：键鼠活跃时间
- `window_monitor.py`：活动窗口标题与切换频率
- `collector.py`：采集快照
- `context.py`：`PerceptionContext`
- `triggers.py`：触发器
- `cooldown.py`：冷却管理
- `engine.py`：主动介入引擎
- `perception_log.py`：触发日志

## 当前输入源

当前实际使用的只有三类：

- 系统时间
- 活动窗口标题
- 用户是否活跃 / 空闲

实现细节：

- `InputTracker` 使用 `pynput`，导入失败时静默降级
- `WindowMonitor` 使用 `pygetwindow`，失败时返回空标题
- 没有截图、OCR、视觉模型调用

## 当前上下文结构

感知层对外只输出 `PerceptionContext`，包括：

- `active_window_title`
- `is_user_active`
- `hour`
- `idle_seconds`
- `switches_per_minute`
- `is_gaming`

人格层只接收这个结构，不直接依赖感知模块内部实现。

## 当前主动触发器

已实现：

- `IdleTrigger`
- `LateNightTrigger`
- `WindowSwitchTrigger`
- `GamingTrigger`
- `LongWorkTrigger` 的逻辑在 `ProactiveEngine` 内联处理

当前规则保持简单：

- 空闲太久
- 深夜仍然活跃
- 切窗口过于频繁
- 游戏窗口粗判
- 持续活跃时间过长

## 当前防打扰策略

通过 `CooldownManager` 做双层冷却：

- 全局冷却：30 分钟
- 同类触发器冷却：1 小时

这保证了主动介入不会变成频繁弹窗。

## 当前触发链路

1. `ConversationService` 初始化可选感知模块
2. 每轮对话前调用 `ProactiveEngine.check()`
3. 若有命中，则返回 `ProactiveEvent`
4. 由 `ConversationService._handle_proactive()` 生成一句角色化台词
5. 触发事件写入感知日志

注意：当前主动介入主要用于 CLI 轮间检查，桌面端还没有独立的后台推送机制。

## 当前没有做的事

下面这些仍然是后续扩展，不是现状：

- OCR
- 截图理解
- 多模态视觉模型
- 高频实时评论用户行为

## 保持简单的约束

- 感知层只输出结构化场景，不生成最终回复
- 触发器只做条件判断，不持有业务状态
- 冷却逻辑独立，不和触发器耦合
- 所有采集失败都允许静默降级

这几个约束要继续保持，不然感知层会很快变成最难维护的部分。
