# 记忆层

当前记忆层的目标很克制：不是尽量记很多，而是给人格层提供一份可控、可裁剪、可解释的上下文。

## 当前模块

- `working_memory.py`：最近若干轮对话
- `summary_memory.py`：会话摘要持久化
- `long_term_memory.py`：结构化长期事实
- `memory_service.py`：统一门面
- `context.py`：`MemoryContext`
- `truncation_log.py`：裁剪日志

## 当前三级结构

### 工作记忆

- 保存在内存中
- 默认保留最近 `10` 轮
- 每轮发送前先 `truncate()`，再追加用户输入

### 摘要记忆

- 存储路径：`data/memories/<character_id>/summaries.jsonl`
- 会话结束时调用 LLM 生成摘要
- 单条摘要最长 `100` 字

### 长期事实记忆

- 存储路径：`data/memories/<character_id>/facts.json`
- 只存结构化 key/value 事实
- 冲突时不直接覆盖，而是标记 `pending_confirm`

## 当前门面设计

应用层只依赖 `MemoryService`，不直接碰底层文件结构。

`MemoryService` 负责：

- 提供 `working_memory`
- 生成 `MemoryContext`
- 会话结束时生成摘要
- 会话结束时提取长期事实

这是当前记忆层最重要的边界，不建议打破。

## 当前注入策略

`get_context()` 会在总 budget 内裁剪记忆：

- 先分配长期事实预算
- 再把剩余预算给摘要
- 只注入已确认事实
- 摘要保持旧到新的顺序

如果有条目被裁掉，会记录到：

- `data/memories/<character_id>/truncation.log`

## 当前写入原则

- 用户明确说出的信息，才进入长期事实
- 长期事实按角色隔离
- 冲突信息不直接覆盖旧值
- LLM 抽取失败时静默降级，不影响主链路

## 当前没有做的事

当前没有：

- ChromaDB
- mem0
- 向量检索
- 关系数据库

所以文档和设计不应再假设记忆层依赖这些组件。

## 后续扩展建议

如果后面需要增强，建议顺序是：

1. 先增强事实抽取和冲突确认
2. 再考虑摘要质量
3. 最后才考虑向量检索

只要 `MemoryContext` 这个输出契约不变，内部存储实现可以演进，但上层不需要跟着重写。
