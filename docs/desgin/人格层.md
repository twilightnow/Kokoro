# 人格层

人格层负责把角色配置、情绪、记忆、感知、关系和安全边界收敛成稳定的 system prompt。它是角色表达的统一出口，不是所有业务状态的存储地。

## 职责

人格层负责：

- 加载和校验角色配置
- 表达角色核心人格
- 维护或接收情绪状态
- 统一拼装 system prompt
- 控制回复风格、口癖、禁用词和行为规则
- 将记忆、感知、关系、安全摘要转成角色可用上下文

人格层不负责：

- 直接读写长期记忆文件
- 判断主动行为是否触发
- 保存关系数值
- 管理提醒和承诺
- 调用 LLM provider
- 执行安全策略最终裁决

## 角色配置

角色配置位于：

```text
characters/<id>/personality.yaml
```

角色配置只描述角色，不描述业务流程。

核心字段类型：

- 基本信息：`name`、`version`、`schema_version`
- 人格核心：恐惧、表层特质、隐藏特质、价值倾向
- 行为规则：回复边界、语气、互动习惯
- 语言特征：口癖、称呼习惯、禁用词
- 情绪表达：mood 到自然语言表达的映射
- 情绪触发：轻量关键词触发规则
- 主动风格：同一主动事件下不同角色的表达方式

角色配置不应写入：

- 用户长期记忆
- 关系状态
- provider key
- 具体运行路径
- 当前会话状态

## PromptContext

人格层对外只接收结构化上下文，不直接依赖其他模块内部实现。

建议 `PromptContext` 包含：

- `character`
- `emotion`
- `memory`
- `perception`
- `relationship`
- `runtime`
- `safety`

其中：

- `memory` 是经过裁剪的记忆摘要
- `perception` 是脱敏后的场景摘要
- `relationship` 是关系状态摘要，不是完整数值表
- `safety` 是安全边界提示，不是可被角色覆盖的建议

## 情绪设计

当前情绪状态可以保持轻量，但长期设计应演进为 `EmotionTimeline`。

情绪状态应包含：

- 当前主要情绪
- 强度
- 原因
- 来源
- 开始时间
- 恢复速度
- 最近变化记录

情绪影响：

- 回复语气
- 表情和动作
- TTS 语速、音量和停顿
- 主动行为倾向

情绪不应直接决定：

- 关系数值大幅变化
- 是否突破安全边界
- 是否强制主动打扰

## 关系输入

关系状态来自 `RelationshipService`，人格层只消费摘要。

可注入内容：

- 当前关系类型
- 推荐称呼
- 熟悉程度
- 最近互动质量摘要
- 用户明确偏好的互动方式
- 当前需要避免的话题

不应注入：

- 完整关系数值历史
- 可被模型随意改写的关系状态 JSON
- 敏感边界全文

## Prompt 组装顺序

建议顺序：

1. 角色身份和核心人格
2. 行为规则
3. 语言习惯和禁用词
4. 安全边界摘要
5. 当前情绪
6. 关系摘要
7. 记忆摘要
8. 当前场景
9. 本轮回复要求

安全边界应靠前注入，并且语义上高于角色设定。

## 人格成长边界

Kokoro 可以表现出“更熟悉用户”，但不应让核心人格随意漂移。

建议分层：

- 核心人格：来自角色配置，稳定
- 关系适应：来自关系状态，缓慢变化
- 共同经历：来自事件记忆，可增长
- 表达偏好：来自偏好记忆，可调整

这样可以让角色有成长感，同时避免长期 prompt 污染。

## 设计约束

- 只有人格层拼 system prompt
- 新上下文优先扩展 `PromptContext`
- 角色 YAML 不写用户状态
- 安全边界不能被角色设定覆盖
- prompt 注入必须受 token budget 控制
- prompt 应表达“她是什么样”，不承载所有业务逻辑
