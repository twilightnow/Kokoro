# 记忆层

记忆层负责把用户历史、共同经历、偏好和边界转成可治理、可裁剪、可解释的上下文。记忆不是越多越好，核心目标是准确、可控、可删除。

## 职责

记忆层负责：

- 保存当前工作记忆
- 保存会话摘要
- 保存长期事实
- 保存偏好、边界和生活事件
- 处理记忆冲突和确认
- 按 token budget 生成 `MemoryContext`
- 为管理界面提供查看、编辑、删除能力

记忆层不负责：

- 生成最终回复
- 判断关系数值
- 决定主动行为
- 判断安全策略
- 存储角色包资源

## 记忆类型

### WorkingMemory

短期对话上下文。

特点：

- 保存在内存中
- 只保留最近若干轮
- 直接参与 LLM messages
- 可被截断

### SummaryMemory

会话级摘要。

特点：

- 保存一段会话的压缩内容
- 用于跨会话延续
- 适合记录“最近聊过什么”
- 不适合存结构化事实和提醒

### LongTermFactMemory

用户明确陈述的稳定事实。

示例：

- 名字
- 职业
- 爱好
- 常用语言
- 明确表达的偏好

设计原则：

- 不写模型推断
- 冲突时进入待确认
- 只注入已确认事实

### PreferenceMemory

用户希望系统如何互动。

示例：

- 喜欢的称呼
- 回复长短
- 提醒频率
- 是否喜欢主动关心
- TTS 声音和语速偏好
- 对某类玩笑是否接受

偏好会影响表达和调度，但不应覆盖安全边界。

### BoundaryMemory

用户明确设定的禁区。

示例：

- 不要记住某事
- 不要再提某人或某话题
- 某类窗口标题不进入感知日志
- 某个时间段不要主动说话

边界记忆优先级高于普通偏好和角色主动风格。

### EventMemory

生活事件和共同经历。

示例：

- 最近压力源
- 正在推进的目标
- 重要日期
- 用户完成过的事
- 共同经历的对话节点

事件记忆应带时间戳、重要性、敏感性和过期策略。

### PromiseMemory

承诺、提醒和待跟进事项。

示例：

- “今晚 11 点提醒我睡觉”
- “明天问我报告交了没”
- “这周别让我连续熬夜”

凡是有时间、重复规则或完成状态的内容，都不应只存在于自然语言摘要中。

## MemoryContext

人格层只消费 `MemoryContext`，不直接读取记忆文件。

建议 `MemoryContext` 包含：

- `recent_summaries`
- `facts`
- `preferences`
- `boundaries`
- `events`
- `promises`

注入原则：

- 边界优先于偏好
- 最近事件优先于旧摘要
- 重要承诺优先于普通事实
- 所有内容受 token budget 控制
- 敏感内容尽量摘要化，不直接原文注入

## 记忆写入流程

推荐写入流程：

1. 从对话或运行时事件中产生记忆候选
2. 判断记忆类型
3. 判断是否敏感
4. 判断是否需要用户确认
5. 写入对应存储
6. 记录来源和更新时间

LLM 可以辅助抽取候选，但不应直接绕过确认和分类逻辑写入长期状态。

## 冲突与确认

冲突处理规则：

- 新值与旧值冲突时，不直接覆盖
- 标记为 pending
- 管理界面显示冲突原因和来源
- 用户选择保留旧值、采用新值、手动编辑或删除

确认机制应覆盖事实、偏好、边界和重要事件。

## 删除与遗忘

用户必须能控制记忆。

需要支持：

- 删除单条记忆
- 删除某类记忆
- 清空某个角色记忆
- “以后不要再提”
- “不要记住这个”
- 导出当前记忆

删除记忆不应删除角色包和静态资源。

## 存储边界

建议运行时数据和角色资源分离：

```text
characters/<id>/        # 角色包和静态资源
data/memories/<id>/     # 记忆数据
data/runtime/<id>/      # 关系、提醒、情绪等运行时状态
```

每类长期数据应有 schema_version，方便迁移。

## 设计约束

- 记忆层只输出结构化上下文
- 不把所有长期信息塞进 facts
- 不把提醒只写进摘要
- 不把边界当普通偏好处理
- 所有长期记忆必须可查看、可编辑、可删除
- 记忆注入必须可裁剪、可解释
