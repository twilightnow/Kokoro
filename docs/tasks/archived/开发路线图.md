## 3. 总体开发原则

- **角色优先**：所有架构决策以"她像她"为最高标准，优先验证人格稳定。
- **本地优先**：用户数据、日志、记忆尽量本地落地，不依赖云端。
- **低打扰优先**：主动介入必须克制，默认不打断用户。
- **分层清晰**：UI 负责展示，感知/记忆/能力只提供输入，人格层负责最终输出。
- **先轻后重**：先做 Level 0 感知和轻量记忆，复杂视觉理解、Live2D、TTS 后置。
- **可迁移**：目录结构、角色配置、日志和记忆数据从一开始就考虑迁移与备份。

## 4. 编码哲学与架构约束

首版实现不追求"功能最多"，而追求"后续还能继续改"。开发时默认遵守以下约束：

- **依赖单向**：`UI -> application/service -> personality -> capability/memory/perception/logger`，底层模块不得反向 import 上层模块。
- **人格层只消费结构化输入**：记忆层、感知层、能力层只返回结构化数据或领域对象，不把 SDK 响应、文件路径、数据库对象直接暴露给人格层。
- **副作用集中**：文件读写、网络调用、系统监听统一放在适配器层；状态机、prompt 组装、触发判断尽量保持纯函数，便于测试。
- **接口先于实现**：在接入新能力前先定义协议或抽象基类，例如 `LLMClient`、`MemoryStore`、`Trigger`、`TTSBackend`，再落具体 provider。
- **组合优于硬编码**：能通过配置、工厂或注册表切换的，不靠 `if provider == ...` 散落在业务代码里。
- **数据模型稳定**：跨层传递统一 dataclass / pydantic model，不直接传 dict 拼接，减少字段漂移。
- **可降级**：记忆、感知、TTS、Live2D 都必须可关闭；任一可选模块失败时，主对话链路仍能工作。
- **先日志再优化**：所有关键决策点先留下可复盘日志，再谈自动化和复杂策略。

### 4.1 模块边界

- `personality`：角色配置、情绪状态、prompt 组装、输出约束。
- `memory`：历史截断、摘要读写、长期事实读写，只提供 `MemoryContext`。
- `perception`：环境采集、触发器判断，只提供 `PerceptionContext` 和 `ProactiveEvent`。
- `capability`：LLM、Embedding、TTS、Vision 等基础能力，只做能力调用与错误封装，返回 `LLMResult` 等领域对象。
- `logger`：会话、感知、成本等日志落盘。
- `application` 或 `service`：编排主流程，连接 CLI / HTTP / UI 与核心模块。

### 4.2 跨层数据契约（Phase 0 必须先行定义）

以下 dataclass 是各层之间传递信息的唯一接口。**Phase 0 就要在对应模块里定义好结构（允许字段暂时为空/占位），后续只在该文件里扩充，不允许跨层直接传 dict。**

```python
# src/capability/llm.py
@dataclass
class LLMResult:
    text: str
    model: str
    provider: str
    input_tokens: int = 0
    output_tokens: int = 0
    finish_reason: str = ""

# src/memory/context.py  （新文件）
@dataclass
class MemoryContext:
    summary_items: list[str] = field(default_factory=list)   # 最近 N 条摘要
    long_term_items: dict[str, str] = field(default_factory=dict)  # key→value 事实

# src/perception/context.py  （已有，Phase 3 前保持最小字段）
@dataclass
class PerceptionContext:
    ...  # 见感知层路线图

# src/perception/event.py  （新文件）
@dataclass
class ProactiveEvent:
    tag: str          # "idle_too_long" / "late_night" / "long_work" 等
    trigger_name: str # 触发器类名，便于日志追踪

# src/personality/prompt_builder.py  （供 prompt_builder 内部聚合参数用）
@dataclass
class PromptContext:
    character: CharacterConfig
    emotion: EmotionState
    memory: MemoryContext = field(default_factory=MemoryContext)
    perception: PerceptionContext | None = None
```

### 4.3 明确禁止的实现方式

- 在 UI、CLI、触发器、记忆模块里各自拼接 prompt。
- 在人格层里直接读写文件、直接调第三方 SDK、直接监听系统事件。
- 在 capability 层里混入角色逻辑、触发条件或记忆写入策略。
- 使用共享全局可变状态串联各层，而不是显式上下文对象。
- 跨层传 raw dict 代替已定义的 dataclass，导致字段漂移无法被类型检查捕获。
- 让文档强绑定某个文件名或单一实现细节，导致一重构文档就失效。

## 6. Phase 0：收口当前 Demo，形成稳定基线

目标：把现有 CLI Demo 收敛成可靠的最小可运行核心，为后续所有阶段提供稳定底座。

重点事项：

- 统一当前项目目录和模块边界
- 抽出 `ConversationService`（`src/application/conversation_service.py`），把 `main.py` 变成薄入口
- 补齐角色配置加载、情绪状态更新、prompt 构建、日志记录之间的主流程
- 定义跨层数据契约（见 §4.2），占位实现即可，但结构必须固定
- 引入 `KOKORO_DATA_DIR` 环境变量（默认 `./data`），`logs/`、`memories/` 路径均基于此计算；**不能等到 Phase 6 才做，Phase 2 落盘就要用**
- 明确 `.env` 配置规则和默认 provider 行为
- 确保 `tests` 至少覆盖核心状态机、配置加载和历史截断
- 整理 README 与设计文档的差异，避免实现与设计脱节

验收标准：

- 连续 10 轮对话不出现明显人格崩坏
- 禁用词控制有效
- 情绪切换符合预期，且能衰减回 `normal`
- `--debug` 和 `--replay` 可用
- 单元测试可通过

## 7. Phase 1：人格层定型

详见 [人格层路线图](人格层_路线图.md)

目标：把"能对话"提升到"角色稳定可控"，让人格层真正成为系统核心。

关键任务：

- 固化角色配置 schema，增加配置校验
- 将 prompt 组装逻辑集中到单一模块
- 给情绪状态机增加可测试的事件输入接口
- 把"人格稳定性"转成可人工复核的日志指标

验收标准：

- 同一角色在不同 provider 下语气不大幅漂移
- 明确触发词输入后，情绪变化可预测
- 人格输出风格能从日志中复核

---

## 8. Phase 2：轻量记忆落地

详见 [记忆层路线图](记忆层_路线图.md)

目标：让角色具备"记得准"的能力，而不是"记得多"的错觉。

关键任务：

- 工作记忆：保留最近 N 轮对话
- 摘要记忆：会话结束后生成摘要并落盘
- 长期记忆：仅写入明确事实（称呼、偏好、长期约定）
- 冲突处理：新事实与旧记忆冲突时先标记，不直接覆盖
- 角色隔离：每个角色单独存储自己的记忆空间

验收标准：

- 角色能记住少量关键信息且不乱编
- 记忆不会无限塞进 prompt
- 角色切换后记忆隔离有效

---

## 9. Phase 3：轻量感知接入

详见 [感知层路线图](感知层_路线图.md)

目标：只接入足够便宜、足够稳定的场景感知，让角色"在合适的时候出现"。

关键任务：

- 感知上下文对象：统一表示窗口、时间、活跃度、场景标签
- 触发器系统：空闲过久、深夜活跃、长时间工作、高频切窗
- 冷却机制：同类事件一段时间内不重复触发
- 人格联动：人格层根据感知上下文决定是否发言及说法

验收标准：

- 主动介入频率受控，不打扰用户
- 相同场景下触发结果稳定
- 触发器可被关闭、调参、复盘

---

## 10. Phase 4：桌面运行时与 UI 骨架

详见 [UI层路线图](UI层_路线图.md)

目标：把 CLI 原型升级为真正的桌面常驻应用，但仍保持"低打扰"和"人格优先"。

关键任务：

- 引入 Tauri 作为桌面壳
- 先做基础悬浮窗、托盘、输入框、气泡，不急着上重表现
- Python 侧作为 sidecar 或本地服务，负责核心逻辑
- 保持 UI 与人格逻辑解耦
- 定义 IPC 协议

验收标准：

- 应用可常驻，不影响正常桌面操作
- UI 可展示当前回复、状态和输入入口
- 核心逻辑仍可独立测试，不被前端绑死

---

## 11. Phase 5：角色表现层增强

详见 [UI层路线图](UI层_路线图.md#phase-5)

目标：在不破坏核心体验的前提下，提升"陪伴感"和"存在感"。

关键任务：

- 静态立绘 + 表情态切换
- 漫画风/轻动画表现尝试
- Live2D 接入验证
- TTS 抽象接口与最小播放流程

验收标准：

- 角色表现增强后不拖慢交互
- 没有表现层时系统仍可运行
- 切换不同表现方案不影响核心对话逻辑

---

## 12. Phase 6：角色包、迁移和扩展机制

详见 [扩展与迁移路线图](扩展与迁移_路线图.md)

目标：让项目从"单角色原型"走向"可安装、可迁移、可扩展的平台"。

关键任务：

- 定义角色 `manifest`，标准化角色目录结构
- 统一用户数据根目录
- 增加导入导出命令（`export` / `import`）
- 角色与记忆打包迁移
- 为触发器、TTS、记忆处理器预留插件接口

验收标准：

- 换机器后可迁移角色、日志和记忆
- 新增角色不需要改核心代码
- 扩展点边界清楚，不污染现有核心模块

## 15. 风险与控制

| 风险 | 描述 | 控制方式 |
|------|------|---------|
| 表现层抢跑 | 过早做 Live2D / TTS，导致核心人格体验未被验证 | 每阶段以"是否更像角色"为验收标准，不以功能数量为主 |
| 记忆/感知过度设计 | prompt 失控、成本失控、行为失控 | 每引入一个新层，保证人格层仍是唯一输出决策中心 |
| UI 先行 | 把项目做成普通聊天壳，而不是低打扰桌面陪伴角色 | 每次扩展都先保证可关闭、可回退、可记录 |
| 模块耦合失控 | 主流程逻辑散落到 CLI、UI、记忆、能力层，后续无法替换实现 | Phase 0 先抽应用编排层与统一上下文对象，禁止跨层偷调 |
| provider 绑定过早 | 先写死单一 SDK，后续切换 OpenAI / Gemini / CLI 成本过高 | capability 层统一抽象接口，provider 通过配置和工厂选择 |

---

## 16. 结论

Kokoro 的正确开发顺序：**先做稳人格核心 → 轻量记忆和感知 → 桌面表现层 → 角色包和扩展生态。**

如果偏离这个顺序，项目很容易在"看起来很丰富"之前，就先失去"角色像不像她"这个最关键的产品价值。
