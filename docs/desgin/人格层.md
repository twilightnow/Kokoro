# 人格层

所属项目：[[Kokoro]]

这是整个项目的核心，也是最需要持续打磨的部分。

设计原则：配置文件负责描述角色，状态机负责管理变化，最终回复始终由人格层统一编排。

---

## 角色配置文件

每个角色对应一个 YAML 配置文件，人格层与代码完全解耦：

```yaml
# characters/<name>.yaml
name: ""
version: ""

personality:
  core_fear: ""       # 核心恐惧/弱点
  surface_trait: ""   # 外在表现
  hidden_trait: ""    # 内在真实

behavior_rules:
  - ""  # 具体行为约束，越精确越好

proactive_style:
  idle_too_long: ""
  user_working_late: ""
  user_gaming: ""

forbidden_words: []   # 破坏人格的词汇
verbal_habits: []     # 口癖
```

建议将人格信息固定为三类，避免 YAML 不断膨胀：

- **基础设定**：身份、核心恐惧、表层特征、隐藏特征、行为规则
- **当前状态**：情绪、亲密度、活跃度、忽视计数
- **表达风格**：口癖、禁用词、称呼方式

约束：
- YAML 只描述角色，不承载复杂流程逻辑
- 复杂判断放到状态机和检索逻辑中
- 人格层统一组装 prompt，其他层不直接拼接角色输出

## 情绪状态机

情绪不依赖模型自由发挥，由外部状态机维护：

```python
class EmotionState:
    mood: str          # angry / normal / shy / happy / cold
    affection: int     # 0-100，好感度
    energy: int        # 0-100，当前状态活跃度
    last_interaction: datetime
    consecutive_ignore: int

def update_state(event: str, current: EmotionState) -> EmotionState:
    # 事件：user_praise / user_ignore / user_late_night /
    #        window_switch / long_idle / user_dependent
    ...
```

每次对话前将当前状态注入 system prompt，对话结束后根据对话内容更新状态。

首版建议只保留最少可验证状态：

- `mood`
- `persist_count`（用于情绪衰减）

`affection`、`last_interaction` 等状态放到后续桌面阶段再引入，避免一开始参数过多却无法调优。

## 台词检索增强（RAG）

```
角色台词数据
    │
    ▼
按情绪标签分类 + embedding 向量化
    │
    ▼
ChromaDB 本地存储
    │
用户输入 → embedding → 检索同情绪标签台词 → 注入 prompt 作为 few-shot 示例
```

台词数据来源因角色而异，需单独整理。

首版约束：

- 没有足够干净的台词库时，不强依赖 RAG
- few-shot 数量保持很少，只作为风格校准
- RAG 不能压过角色基础设定和当前状态
