## 模块职责

- **负责**：对话历史管理、会话摘要生成与存储、长期记忆读写
- **不负责**：决定记忆内容如何影响角色输出（由人格层决定）
- **约束**：记忆只向人格层提供输入，按角色隔离存储

## 三级记忆结构

```
长期记忆
├── 用户基本信息（名字、职业、习惯）
└── 关键关系节点（第一次对话、重要事件）

摘要记忆
├── 历史对话摘要（每次会话结束后生成）
└── 承诺/约定记录

工作记忆
└── 最近 10 轮对话（运行时维护，不落盘）
```

## Phase 0：工作记忆收口

目标：确保工作记忆在主流程中正确截断，不让 prompt 无限增长。

### 任务清单

- [ ] 确认 `working_memory.py` 的截断逻辑（`max_rounds=10`）在每轮发送前执行
- [ ] 截断时从最老的轮次开始删除，保留最新 N 轮
- [ ] 写单元测试：超过 10 轮后，历史长度不超过上限

### 验收

- 连续 20 轮对话后，传入 LLM 的 messages 列表长度不超过 21（10 轮 × 2 + system）

---

## Phase 2：轻量记忆落地

### 2.1 会话摘要记忆

目标：会话结束后生成摘要，作为下次会话的上下文背景。

技术方案：

- 会话结束时调用 LLM 生成摘要（50-100 字），写入 JSONL 文件
- 摘要按角色隔离存储：`memories/<character_id>/summaries.jsonl`
- 下次会话开始时，加载最近 N 条摘要，注入 system prompt 的摘要区域

任务：

- [ ] 在 `main.py` 的会话结束钩子处调用摘要生成
- [ ] 实现 `src/memory/summary_memory.py`：`generate_summary(history) -> str`，`save_summary(character_id, summary)`，`load_recent_summaries(character_id, n=3) -> list[str]`
- [ ] 在 `prompt_builder.py` 中增加 `summary_context` 注入点（位于 system prompt 的记忆区域）
- [ ] 写单元测试：给定对话历史，验证摘要生成不为空且长度合理

### 2.2 最小长期记忆

目标：只记录用户明确说出的事实，不乱记。

存储方案（首版用 JSON 文件，不依赖 ChromaDB）：

```json
// memories/<character_id>/facts.json
{
  "user_name": "值",
  "user_occupation": "值",
  "preferences": [],
  "agreements": []
}
```

写入原则：

- 用户明确说出的事实，才进入长期记忆
- 模型推断出的内容，默认只进入摘要记忆
- 发生冲突时保留旧记录并标记 `pending_confirm: true`，不直接覆盖

任务：

- [ ] 实现 `src/memory/long_term_memory.py`：`read_facts(character_id)`，`write_fact(character_id, key, value)`，`flag_conflict(character_id, key, new_value)`
- [ ] 在 `prompt_builder.py` 中增加 `long_term_context` 注入点（精简格式，不超过 200 tokens）
- [ ] 写单元测试：连续写入同一 key 两次不同值，验证冲突标记而非覆盖
- [ ] 确保每个角色的记忆存储路径隔离，不同角色不共享 facts.json

### 2.3 记忆注入长度控制

目标：记忆内容注入 prompt 时不能无限增长。

任务：

- [ ] 摘要记忆：只注入最近 3 条，每条不超过 100 字
- [ ] 长期记忆：只注入高权重事实，总计不超过 200 tokens
- [ ] 在 `prompt_builder.py` 中增加 token 预算检查，超过阈值时截断记忆上下文

### 验收

- 下次会话开始时，prompt 中包含上次摘要
- 用户说"我叫 XX"后，facts.json 中写入用户名称
- 角色切换后，各自使用独立的 summaries.jsonl 和 facts.json
- 记忆注入后 prompt 总长度不超过设定阈值

## Phase 3 之后：ChromaDB / mem0（后置）

前置条件：JSON 文件方案已稳定，长期记忆写入原则已验证。

任务（后置）：

- [ ] 引入 ChromaDB，按角色 ID 分 collection 存储向量化记忆
- [ ] 实现语义检索：根据用户输入检索相关长期记忆片段
- [ ] 评估是否引入 `mem0` 做自动记忆提取（谨慎，避免乱记）
- [ ] SQLite 存储关系状态（亲密度、忽视计数等，Phase 4 之后）

## 依赖说明

| 依赖 | 来源 | 说明 |
|------|------|------|
| 角色 ID | 人格层 | 用于记忆隔离的目录命名 |
| 摘要生成 LLM 调用 | 能力层 | 会话结束时调用一次 |
| Prompt 注入点 | 人格层 | `prompt_builder.py` 需预留接口 |

---

## 不建议在此阶段做的事

- Phase 2 之前引入 ChromaDB（JSON 文件足够，不需要向量数据库）
- 自动从模型输出中提取记忆（容易乱记，先用明确事实写入）
- `mem0` 在验证写入原则之前引入
